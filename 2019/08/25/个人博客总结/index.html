个人博客总结

发表于 2019-08-25 | 分类于 技术 | | 阅读次数:

一.前言：

  从 2019.7.16 到今天 2019.8.24 加入新蛋也一个多月时间了，这一个月成长了很多，从技术、思维、心态都从1个月之前的小小白晋升到了现在的小白啦。这一切都要感谢公司，对于一个目前还 “毫无价值” 的实习生的用心用力的培训。

二.技术：

1.前端：Angular CLI version: 8.2.0
  (1).Angular Material 提供封装好的组件支持，超级好用，博客很多地方使用到 卡片组件。满满调包的味道。使前端开发变得极为简单。
  (2).HttpClient 作为前端和后端交互的必要工具，数据格式使用的比较通用的 json。
  (3).markdown 使用的 editor markdown编辑工具，集成到 Angular 相当方便。编辑页面简单大方。
  (4).总结：这次前端选择 Angular 的原因在于培训的时候就是用的 Angular.因为前端本身是我的一个弱项，一直想提升，所以我是很愿意去学好前端的，而这次又给了我这个机会，让我对 Angular 加深了影响，现在还只是浅显的使用，以后如果选择纳入自己的技术栈的话一定需要去深入。
2.后端：ASP .net core API version:2.1
  (1).JWT JSON Web Token的简称。token的意思是 象征、证明、记号的意思。使用 JWT 就可以验证是哪个用户用哪个账号能操作什么。
  (2).EF Entity Framework的简称。很好用，屏蔽了 sql 语句。但是复杂业务可能并不能去依靠 EF。还要自己写出足够的更符合业务的、更高效的 sql 语句。
  (3).log4net 处理业务日志。其实日志我一直不是很会处理，更多是输出一些日志的 info.当成调试日志来处理了。以后一定要多思考业务事先应将什么内容纳入日志文件。
  (4).总结：其实如果使用 Java 框架来开发这次的 api 的话我可能能把这个后台写的更加健壮。但是我们部门主要的语言是C#，所以我就使用了 .net core 框架。因为 C# 本身我只学习了基础语法，还没有深入。所以很多业务我不得不考虑用自己的方式去实现而不能用C#高级语法所支持的。这次我用这个框架模拟了一个我认为的 MVC 的架构模式去开发，因为涉及不深，很多时候以及模糊了语言差异，所以比较顺利的完成了 API conding。
3.测试工具：Postman
  (1).主要是使用 Postman 来测试请求，看结果是否符合自己的预期。算是对自己 API 的一个健壮性的测试。很严重的是，这次博客作业我并没有对API进行细粒度的测试，更没写测试用例，后果就是测试博客功能的时候，很多问题就争相涌现，导致不停的修修补补，东一块西一块，这本身对整个前后端的代码会造成一定的影响。

三. 作业流程:

  需求文档 => 接口文档 => 数据库设计+实现 => 后台API设计+实现 => 前台页面设计+实现 =>数据交互。首先，我在需求文档上花的功夫太少了，写的很差！！导致后面的连锁反应，接口文档出现问题、conding出现问题。还有这个流程的顺序也有很大问题，不应该在页面设计出来之前就开始写 API，这样导致的后果就是在交互的时候 api 和页面想呈现的内容不完全一致。这样的流程也不是一定存在问题，前提是需要相当的确定api给的数据一定是前台所需要的。所以我认为这个 coding 的流程不应该是从 后台 => 前台 这样的开发方式了，因为这样需要太多的思考及相当的自信。而应该根据业务来 coding，先设计页面、再写 api，最后进行交互，这样如果是出现问题也会更细粒度了。就比如说这次的一个返回json问题。因为我后台不完全返回的是 json，可能在没取到数据的情况下我就返回的 “xxxId不存在” 这样的描述信息。这样的一个后果就是我 angular 比较陌生不知道怎么拿呀！如果按照业务的方式来的话我就会立马知道是应该去建立一个通用的json返回类了。还有就是应该建立一个大局观，这不局限于去做一个博客，应该是做每一件事情。比如技术选型，产出内容，页面排版，从整个整体到局部的整体都应该事先考虑好，而不是说走一步算一步，这样风险太大，做不到提前规避。最后，重要的是这次遇见的问题发现了下次就一定要去避开这类同样的问题，而不是继续去走重复的路，这才能成长、得到更好的提升。

四.功能

  大作业要求的功能都完成了。只是评论只做到单一评论，不能回复。
  登录。登录主要是提升权限。这次博客是个人博客，只有我自身一个人是管理员，所以其实 JWT 只是给了我一个 token 而已。后面判断登录状态都只需前端判断 token 存在与否，不存在后端做一层验证逻辑。
  分类 => 编辑、添加、删除。
  markdown编辑文章。
  一些小卡片：个人资料、个人分类、阅读排行。

五.总结

  总体来说，这次的大作业我要给自己打一个差评。文档、规范、代码思想、欠缺思考都出现在了这次写博客的过程中。从呈现上来看，我还是很满意博客的样子的，比我预期的要好很多，准备以后再好好修改、添加一点内容，当成自己的主站。博客作业是从 8/9 到 8/24 的一个时间段。其实感觉还是蛮短的，因为整个流程还是比较多，这对于业务经验比较少的我来说只有舍弃一些东西以加快进度，比如减少一些思考，因为有时候用到一个技术想去了解前因后果再很有逻辑性的去应用它，而这次我就复制粘贴看效果，有效果就继续没效果就不断找文章继续复制粘贴，对于经验比较少的我来说还是太急躁了，因为太多的东西我不能去预估，减少一定的思考是怕完不成工作不得已的一个选择。而思考对于我来说，我感觉又是比较重要的，所以后面我需要给自己时间去学习很多东西，而不是在上班的时候再追赶进度，这样才能有效得到提升。
